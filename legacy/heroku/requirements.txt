# Gunicorn — installed but NOT the current entrypoint.
#
# Why it's here: standard Heroku Python deploys use gunicorn, and
# pinning it keeps the slug consistent with that convention. A
# future migration to WSGI (Flask wrapper around the existing routes,
# or FastAPI rewrite) drops in without a fresh `pip install`.
#
# Why it's NOT in the Procfile: rcm_mc.server is stdlib
# http.server.ThreadingHTTPServer — not a WSGI app. The Procfile
# launches it directly via `python -m web.heroku_adapter`. Pointing
# gunicorn at the stdlib server would fail at startup — don't switch
# the Procfile to gunicorn until the WSGI migration actually lands.
gunicorn>=21.0,<24.0
