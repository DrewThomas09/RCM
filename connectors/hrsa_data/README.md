# connectors/hrsa_data — HRSA data downloads (HPSA / MUA / Health Center sites)

A self-contained, **stdlib-only** connector that ingests HRSA's public
data-download CSVs into canonical SQLite tables and re-exposes them
behind the estate's uniform `/v1` query surface. Mirrors the
architecture of `connectors/cms_coverage` exactly.

## Source: CSV downloads, not a JSON API

HRSA publishes these datasets as stable CSV files under
`https://data.hrsa.gov/DataDownload/DD_Files/{NAME}.csv` (the human
index is https://data.hrsa.gov/data/download). There is no key, no
query API and no paging — each dataset is one 10–60 MB file. All five
file names below were **verified live on 2026-07-06** (HTTP 200 +
expected header row; no alternate names were needed):

| File | Size | Rows | Dataset |
|------|------|------|---------|
| `BCD_HPSA_FCT_DET_PC.csv` | ~48 MB | ~79k | `hpsa_primary_care` |
| `BCD_HPSA_FCT_DET_DH.csv` | ~28 MB | ~46k | `hpsa_dental` |
| `BCD_HPSA_FCT_DET_MH.csv` | ~25 MB | ~41k | `hpsa_mental_health` |
| `MUA_DET.csv` | ~11 MB | ~20k | `mua` |
| `Health_Center_Service_Delivery_and_LookAlike_Sites.csv` | ~14 MB | ~19k | `health_center_sites` |

If HRSA renames a file, the transport raises a clear 404 error pointing
at the download index; the fix is a one-line `file_name` edit in
`endpoints.py`.

The files are regenerated by HRSA's data warehouse frequently (the
sampled files carried a warehouse create date three days old); the
registry advertises a conservative `monthly` refresh cadence.

## Datasets

| dataset_id | Target table | Slice | Natural key |
|------------|--------------|-------|-------------|
| `hrsa_data_hpsa_primary_care` | `hrsa_hpsa` | `source_endpoint='hpsa_primary_care'` | `{hpsa_id}:primary_care:{geo_id}` |
| `hrsa_data_hpsa_dental` | `hrsa_hpsa` | `source_endpoint='hpsa_dental'` | `{hpsa_id}:dental:{geo_id}` |
| `hrsa_data_hpsa_mental_health` | `hrsa_hpsa` | `source_endpoint='hpsa_mental_health'` | `{hpsa_id}:mental_health:{geo_id}` |
| `hrsa_data_mua` | `hrsa_mua` | — | `{mua_p_id}:{service_area}:{county_fips}:{subdiv_fips}:{tract}:{component}` |
| `hrsa_data_health_center_sites` | `hrsa_health_center_sites` | — | `{bphc_assigned_number}` |

The three HPSA discipline files share one table (their column sets are
identical except `pc_mcta_score`, which only the primary-care file
publishes); the registry's `source_filter` pins each dataset to its
slice, and the discipline token inside the key makes cross-file
collisions impossible.

**Key validation (full live files, 2026-07-06):** every composed key is
unique except a handful of *byte-identical duplicate lines* HRSA ships
(5 in PC, 98 in DH, 28 in MUA) — the idempotent upsert collapses those
harmlessly. MUA/P ids are re-used across designations (a Designated and
a Withdrawn designation can share an id *and* a geography), which is
why the service-area name is part of the MUA key.

## Columns

CSV headers are prose ("MUA/P ID", "% of Population Below 100%
Poverty"); `normalize.snake()` is the one documented header→column rule
(`%` → `pct`, other punctuation runs → `_`), and the column lists in
`tables.py` are snapshots of the live headers passed through it. **All
published columns are kept**, TEXT-typed (SQLite is dynamically typed;
the query layer casts explicitly). Every live file ends its header and
data rows with a dangling comma — the transport drops that empty
trailing cell. Values are whitespace-stripped at normalize time (the
live files pad some numerics, e.g. `" 6"`).

A column HRSA adds later is not dropped silently: it lands in the
`unmapped_fields` audit that `refresh()` returns.

## Size guardrail

`fetch` defaults to `max_rows=50_000` per file (the CLI's `--full`
lifts it). The transport **streams** the response, so a capped fetch
downloads only what it parses — `--max-rows 500` moves ~500 rows of
bytes, not 48 MB.

## Usage

```bash
python -m connectors.hrsa_data.cli discover
python -m connectors.hrsa_data.cli datasets

# Ingest (capped / full)
python -m connectors.hrsa_data.cli fetch --dataset hpsa_primary_care --max-rows 500
python -m connectors.hrsa_data.cli fetch --dataset mua --full

# Query the canonical tables (uniform estate grammar)
python -m connectors.hrsa_data.cli query hrsa_data_hpsa_primary_care \
    --filter common_state_abbreviation=TX --filter hpsa_score__gte=18 \
    --sort -hpsa_score --limit 10

# State-level lookups
python -m connectors.hrsa_data.cli lookup-shortage-area TX
python -m connectors.hrsa_data.cli lookup-health-center TX

# Standalone /v1 surface
python -m connectors.hrsa_data.cli serve --port 8099
```

### `/v1` routes

```
/health
/v1/datasets
/v1/query/{dataset}?<filters>&select=&sort=&limit=&offset=
/v1/query/{dataset}/aggregate?group_by=a,b&<filters>&limit=
/v1/lookup/shortage-area/{state}[?limit=N]   HPSA + MUA picture for a state
/v1/lookup/health-center/{state}[?limit=N]   Health Center Program footprint
```

Filter grammar (uniform across the estate): `field=v` equality or
`field__op=v` with `eq/ne/gt/gte/lt/lte/like/in/between/isnull/notnull`;
sort `field` / `-field`.

## Join keys

- `hrsa_hpsa` / `hrsa_mua` → state + county FIPS (joins to anything
  geographic).
- `hrsa_health_center_sites.fqhc_site_npi_number` → joins to the NPI
  Registry connector's provider tables.

## Rate limits

None published for the download files; the transport keeps a 1 s
inter-request floor and 429/5xx exponential backoff with `Retry-After`
support as a courtesy posture, not a documented contract. Each request
is a whole file, so traffic is naturally tiny — but a `--full` pull of
all five files moves ~125 MB; schedule accordingly.

## Tests

```bash
cd /home/user/RCM
python3 -m unittest discover -s connectors/hrsa_data/tests -t . -v
```

No test touches the network: the transport's opener is injectable and
the fixtures in `tests/fakes.py` reproduce the live files' exact header
rows and quirks (dangling trailing comma, padded values, quoted commas,
duplicate lines, BOM tolerance). Live smoke is manual:
`python -m connectors.hrsa_data.cli fetch --dataset hpsa_primary_care --max-rows 500`.
